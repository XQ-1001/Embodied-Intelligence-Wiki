# CLAUDE.md — 具身智能Wiki 工作规范

## 一、信息来源处理规范

### 1.1 网站/URL 身份验证协议

**问题**：此前因 WebSearch 未搜到某 URL 与公司的关联，就错误地否定了该 URL 是该公司官网，忽略了用户的直接指认。

**规范**：
- **WebSearch 用于发现，WebFetch 用于验证**。搜索不到不等于不存在。
- 当用户指出某个 URL 属于某公司时，**必须先直接用 WebFetch 访问该 URL**，查看页面内容确认，而不是依赖搜索结果的间接证据。
- 如果 WebFetch 被阻止，先排查配置问题（如 `~/.claude/settings.json` 中的 `skipWebFetchPreflight`），而非直接给出否定结论。
- 当用户提供的信息与搜索结果的"缺失"矛盾时，优先信任用户的直接知识。

### 1.2 引用来源必须实际读取

**问题**：将播客列为信息源，但从未实际读取/转写其内容，导致遗漏关键数据（融资总额）。

**规范**：
- 每个列在"参考资料"中的来源，**必须实际读取其内容**，不能仅凭摘要或简介就引用。
- 对于播客/音频类来源：必须下载音频并通过语音转写获取文字内容，不能仅引用 shownotes。
- 对于网页来源：必须用 WebFetch 直接获取页面内容，不能仅依赖搜索摘要。

### 1.3 多方交叉验证

- 同一事实尽量从多个独立来源验证（官网 + 媒体报道 + 工商信息 + 播客等）。
- 标注每条事实的来源可靠性（如"公司自述""媒体报道""工商登记""播客对话"等）。

## 二、音频素材处理（播客/访谈/会议录音）

### 2.1 处理流程

当需要从播客或音频中提取信息时，遵循以下流水线：

```
1. 获取音频 URL → 从播客页面 JSON 数据中提取 .m4a/.mp3 链接
   小宇宙FM: __NEXT_DATA__ → props.pageProps.episode.enclosure.url
   earsonme.com: 通常在 HTML 的 JSON-LD 或媒体元素中

2. 下载音频 → curl -sL -o /tmp/audio.m4a <URL>

3. 转写 → faster-whisper（已安装在环境中）
   - 短音频（<5分钟）：直接用 base 模型
   - 长音频（>5分钟）：先用 vad_filter=True 加速，或分段处理
   - 中文优先使用 language="zh"

4. 提取目标信息 → 从转写文本中搜索关键词
```

### 2.2 可用工具

- `faster-whisper`：本地语音转写（pip 已安装）
- `ffmpeg`：音频格式转换、裁剪（已安装）
- 模型选择：`tiny`（极速，质量一般）、`base`（推荐，速度/质量平衡）

### 2.3 注意事项

- 首次运行 `faster-whisper` 需要下载模型，模型会缓存到 `~/.cache/huggingface/`
- 若单个 bash 命令超时（45s限制），将长音频切分为 1-3 分钟的小段处理
- WAV 分段时务必使用 `-acodec pcm_s16le -ar 16000 -ac 1` 参数重新编码，不要使用 `-c copy`
- 中文转写使用 `language="zh"` 参数可获得更准确结果

## 三、WebFetch 配置与故障排查

### 3.1 预检（preflight）检查

如果 WebFetch 返回被阻止的错误：
1. 检查 `~/.claude/settings.json` 是否存在
2. 如果不存在，创建该文件并添加：
   ```json
   {
     "permissions": {
       "allow": ["WebFetch", "WebSearch"],
       "skipWebFetchPreflight": true
     }
   }
   ```
3. 配置后重试 WebFetch

### 3.2 已知可正常访问的站点

默认情况下可直接访问的站点包括但不限于：科技媒体、主流新闻网站、公司官网（非中国大陆ICP限制的域名）、GitHub、Wikipedia、播客平台页面等。如果特定域名被阻止，优先通过配置解决，不要放弃获取信息。

## 四、错误复盘机制

### 4.1 当用户指出错误时

1. 立即承认错误，不要争辩或找借口
2. 追溯错误的根本原因（不是"我搞错了"，而是"我犯了什么逻辑错误/流程错误"）
3. 将原因和解决方案记录到 CLAUDE.md 中
4. 验证修复是否有效

### 4.2 常见错误模式（来自历史复盘）

| 错误 | 根因 | 解决方案 |
|------|------|---------|
| 否定官网身份 | 用"搜索没找到"替代"直接访问验证" | 1.1 协议 |
| 播客信息遗漏 | 列了来源但没实际读取 | 1.2 规范 + 第二章流程 |
| 数据无来源 | 引用公司/媒体数据时未标注出处 | 1.3 交叉验证 |

## 五、Wiki 数据质量标准

- 每条事实必须在页面中标注来源编号（`<sup><a href="#refN">[N]</a></sup>`）
- 同一事实在页面中只有一处规范位置
- 公司自述数据与独立验证数据要区分标注
- 模糊/不确定的数据需在分析层标注审慎提示
- 更新页面后同步更新 index.html（排序、编号、统计、标签）和 tracker.csv
